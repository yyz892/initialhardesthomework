## 13.1 项目结构设计

### 为什么采用三层架构？

```
src/
├── model/              数据模型层 → 只存数据，不管业务
├── service/            业务逻辑层 → 只处理逻辑，不管界面
├── util/               工具层 → 通用功能
└── GameApp.java        入口 → 界面+调度
```

**为什么不把所有代码写一个文件里？**

```java
// ❌ 全部塞一个文件：几百行代码搅在一起
public class AllInOne {
    // User类 + 注册 + 登录 + 验证码 + 英雄 + 敌人 + 战斗 + 菜单
    // 改一个地方可能影响其他地方，找代码像大海捞针
}

// ✅ 分层分文件：每个类职责单一
// User.java       → 只管用户数据
// UserService.java → 只管注册登录逻辑
// BattleService.java → 只管战斗
// 哪出问题改哪，思路清晰
```

**分层的核心原则：单一职责**。一个类只做一件事，做好它。

## 13.2 用户系统设计

### 为什么User要设计为JavaBean？

```java
public class User {
    private String username;   // 私有 → 防止外部随意修改
    private String password;

    public User() {}            // 无参 → 框架/工具需要
    public User(String u, String p) { this.username = u; this.password = p; }

    // get/set → 受控访问
    public String getUsername() { return username; }
    public void setUsername(String username) { this.username = username; }
}
```

**设计决策**：
- **密码私有化**：不让外部随便读取密码（安全考虑）
- **无参+有参构造都写**：无参用于框架反射，有参用于方便创建
- **get/set标准写法**：IDEA一键生成，行业标准

### 为什么用ArrayList存用户而不是数组？

```java
// 注册用户数量不确定 → 必须用ArrayList
private static ArrayList<User> userList = new ArrayList<>();

// 如果用数组：
User[] users = new User[100];  // 最多100个，多了存不下
// 而且每次注册都要记录当前存到第几个了，麻烦
```

## 13.3 验证码设计

### 为什么验证码要字母+数字混合？

```java
public static String generate(int length) {
    char[] pool = new char[62];
    // 26个大写 + 26个小写 + 10个数字 = 62种可能
    // 4位验证码 = 62⁴ ≈ 1477万种组合，足够安全
    Random r = new Random();
    StringBuilder sb = new StringBuilder();
    for (int i = 0; i < length; i++) {
        sb.append(pool[r.nextInt(pool.length)]);
    }
    return sb.toString();
}
```

**设计决策**：
- **不用纯数字**：容易被暴力破解，4位数字只有1万种
- **不用纯字母**：用户体验不好，有些人觉得输字母麻烦
- **混合型**：安全性够，体验尚可，适合学习项目
- **为什么用StringBuilder**：循环中拼接字符，避免创建多个String对象

## 13.4 角色继承体系

### 为什么用抽象类而不是普通类？

```java
// 抽象父类：不能直接new，强制子类实现攻击方法
public abstract class GameCharacter {
    protected String name;
    protected int hp, maxHp, attack, defense;

    // 抽象方法 → 每个角色攻击方式不同，父类无法给出默认实现
    public abstract int normalAttack(GameCharacter target);
    public abstract int skillAttack(GameCharacter target);

    // 具体方法 → 受伤逻辑所有角色都一样，写一份共享
    public void takeDamage(int damage) {
        int real = damage - defense;
        if (real < 1) real = 1;
        hp -= real;
    }

    public boolean isDead() { return hp <= 0; }
}
```

**为什么不用接口？**
- 角色有大量共享代码（受伤、血条、属性计算）
- 如果用接口，这些代码要在每个实现类都写一遍
- 接口更适合纯"能力"定义（如Flyable），不适合有大量共性代码的场景

### 为什么Hero和Enemy要分开两个类？

```java
// Hero：玩家操作的英雄
public class Hero extends GameCharacter {
    private int score;              // 积分（敌人没有）
    private int skillMpCost;        // 技能消耗（敌人没有这个概念）

    public void recoverAfterBattle() { }  // 战后恢复（英雄独有）
    public void enhanceAttributes() { }   // 属性增强（英雄独有）
}

// Enemy：AI控制的敌人
public class Enemy extends GameCharacter {
    private EnemyType type;        // 敌人类型（英雄没有）
    private int scoreValue;        // 击败分值（英雄没有）

    public int aiAttack(GameCharacter target) { } // AI决策（英雄没有）
}
```

**如果强行用一个类**：会有大量if判断，类会臃肿不堪，违背单一职责原则。

## 13.5 战斗系统设计

### 为什么战斗是回合制？

```java
public static boolean startBattle(Hero hero, Enemy enemy) {
    while (!hero.isDead() && !enemy.isDead()) {
        // 1. 英雄回合 → 玩家选择行动
        // 2. 敌人受伤 → 检查死亡
        // 3. 敌人回合 → AI决策
        // 4. 英雄受伤 → 检查死亡
    }
}
```

**回合制的优点**：
- 逻辑清晰，英雄和敌人交替行动
- 适合控制台交互（等玩家输入）
- 状态可控，每一步都知道发生了什么

### 为什么敌人攻击用AI随机决策？

```java
public int aiAttack(GameCharacter target) {
    Random r = new Random();
    return r.nextInt(100) < 40
        ? skillAttack(target)    // 40%概率放技能
        : normalAttack(target);  // 60%概率普攻
}
```

**设计决策**：
- 如果敌人总是普攻 → 太无聊
- 如果敌人总是大招 → 太难
- 随机使每场战斗有变化，增加趣味性和不确定性

## 13.6 血条可视化

```java
public void printHealthBar() {
    int barLength = 20;
    int filled = (int)((double)hp / maxHp * barLength);  // 按比例计算填充数
    StringBuilder sb = new StringBuilder("[");
    for (int i = 0; i < barLength; i++) {
        sb.append(i < filled ? "█" : " ");
    }
    sb.append("] " + hp + "/" + maxHp);
    System.out.println(sb);
}

// 效果：
// 满血：[████████████████████] 200/200
// 半血：[██████████          ] 100/200
// 残血：[███                 ] 30/200
```

**为什么用比例计算？**
- 不同角色血量不同（100~300），统一用比例才能在20格内可视化
- `(当前HP / 最大HP) × 20` = 应该填满多少格

## 13.7 积分段位系统

```java
// 击败敌人获得积分：等级 × 100
// 段位评定：
String getRank(int score) {
    if (score >= 1000) return "传说勇者";
    else if (score >= 600) return "黄金勇士";
    else if (score >= 300) return "白银战士";
    else if (score >= 100) return "青铜冒险者";
    else return "新手冒险者";
}
```

**设计决策**：
- 积分=等级×100：越强的敌人越值钱，鼓励挑战高难度
- 段位制：给玩家明确的目标感和成就感
- if-else从高到低判断：利用else if的短路特性

## 13.8 设计决策总结

| 你在项目中做了什么 | 为什么这样设计 |
|------------------|--------------|
| 三层架构(model/service/util) | 单一职责，好维护 |
| User用JavaBean | 行业标准，封装安全 |
| 用ArrayList存用户 | 用户数量不确定 |
| 用数组做验证码字符池 | 固定62个字符 |
| 用抽象类做角色父类 | 有共享代码要继承 |
| 用枚举定义敌人类型 | 有限的固定类型 |
| 用接口做能力扩展 | Flyable/Swimable可选 |
| 战斗用while循环 | 不确定打几回合 |
| 不用数据库 | 学习阶段未涉及，用内存集合足够 |
| 不用GUI只做控制台 | 专注Java核心，GUI不是当前目标 |
